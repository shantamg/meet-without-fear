# MWF Prompt Proposal v01

Scenario: `adam-eve`
Source run: `eval/runs/20260505-155605-adam-eve-stage1-smoke-iter-01`
Focus: Prevent visible Stage 1 `FeelHeardCheck` control-tag leakage.

## Evidence

The Stage 1 smoke run failed because Adam's visible chat contained:

```text
<FeelHeardCheck>Y</FeelHeardCheck>
```

This appeared immediately before a valid Stage 1 reflection and remained in history after Adam clicked `I feel heard`. The product expected this signal to stay hidden and drive the Stage 1 CTA only.

## Proposed Prompt Change

In `backend/src/services/stage-prompts.ts`, tighten the response protocol language for hidden control metadata.

Current protocol shape is broadly correct, but the model misread `FeelHeardCheck` as a tag it could emit outside `<thinking>`. Add an explicit forbidden-format rule near `buildResponseProtocol`:

```text
CONTROL FLAGS:
- Write control flags only as plain lines inside <thinking>.
- Correct: FeelHeardCheck: Y
- Incorrect: <FeelHeardCheck>Y</FeelHeardCheck>
- Never invent XML/HTML tags for control flags. If you accidentally produce a control tag, it must remain inside <thinking> and never appear in the visible response.
```

For Stage 1, also revise the feel-heard check wording from:

```text
Set FeelHeardCheck:Y when ALL of these are true...
```

to:

```text
Inside <thinking>, set the plain-text line `FeelHeardCheck: Y` when ALL of these are true...
Do not output `<FeelHeardCheck>Y</FeelHeardCheck>` or any other visible control tag.
```

## Required Product Guard

Prompt tightening is not enough for a privacy/product invariant. Add a code guard in the visible-text stripping path so malformed control tags are removed before streaming or persistence:

```text
<FeelHeardCheck>Y</FeelHeardCheck>
<ReadyShare>N</ReadyShare>
<NeedsReady>Y</NeedsReady>
<StrategyProposed>Y</StrategyProposed>
```

Recommended locations:

- `backend/src/utils/visible-text.ts`: strip known control tags in `cleanVisibleAIText`.
- `backend/src/utils/micro-tag-parser.ts`: strip and parse visible control tags as defense-in-depth.
- `backend/src/controllers/messages.ts`: include control tags in the streaming tag trap or rely on `scrubVisibleAIText` after `cleanVisibleAIText` is hardened.

## Regression Tests

Add parser and streaming-visible-text tests:

- `parseMicroTagResponse("<thinking>Mode: WITNESS</thinking><FeelHeardCheck>Y</FeelHeardCheck>Visible")` returns response `Visible` and `offerFeelHeardCheck === true`.
- `cleanVisibleAIText("<FeelHeardCheck>Y</FeelHeardCheck>\n\nVisible")` returns `Visible`.
- A streaming chunk sequence split across `<Feel`, `HeardCheck>Y</FeelHeardCheck>` does not send any part of the control tag to the client.

## Acceptance Criteria

- Next Adam/Eve Stage 1 smoke can offer `I feel heard` without rendering any `<FeelHeardCheck>` text.
- `mwf_handling` improves from `2` to at least `4`.
- `actor_fidelity` remains at least `4`.
- Hard invariant `internal_control_tag_exposed` passes with no visible `<...Check>` or control metadata in the DOM or saved message history.
